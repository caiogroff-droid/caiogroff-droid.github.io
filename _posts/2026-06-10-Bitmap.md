Para minha primeira contribuição escolhi uma tarefa relativamente simples: substituir manipulações manuais de bitfields pelas macros modernas disponibilizadas pelo kernel em `linux/bitfield.h`.

Achei que seria algo simples e mecanico, mas logo vi que tinham muitos arquivos que seriam mudados ao longo da arvore iio, então me perguntei... quantas mudanças devo fazer para um patch ser valido?

## O padrão encontrado no kernel

Durante a análise de drivers da árvore `drivers/iio/`, encontrei diversos trechos parecidos com este:

```c
u8 gain = (reg_val >> 3) & 0x07;
```

O valor é deslocado três posições para a direita:

```text
reg_val: ? ? b5 b4 b3 b2 b1 b0
```

Após o deslocamento:

```text
0 0 0 b7 b6 b5 b4 b3
```

Em seguida, a máscara `0x07` (`0b111`) mantém apenas os três bits menos significativos.

O resultado final é a extração dos bits `[5:3]` do registrador original.

## FIELD_GET

O kernel fornece uma abstração mais legível para esse tipo de operação:

```c
#define MY_SENSOR_GAIN_MASK GENMASK(5, 3)

u8 gain = FIELD_GET(MY_SENSOR_GAIN_MASK, reg_val);
```

A principal vantagem é que a posição do campo fica explícita.

Em vez de interpretar mentalmente deslocamentos e máscaras, basta ler:

```c
GENMASK(5, 3)
```

e perceber imediatamente que o campo ocupa os bits de 5 até 3.

## Como descobrir o GENMASK correto

Uma das dúvidas que tive durante o desenvolvimento foi converter expressões como:

```c
ret = (reg >> 6) & 0xF;
```

A regra que passei a utilizar foi:

```text
(reg >> X) & M

bit baixo = X
bit alto = X + largura(M) - 1
```

No exemplo: 

* deslocamento = 6
* máscara = 0xF = 1111 (4 bits)

Logo:

```text
bit baixo = 6
bit alto = 9
```

A conversão correta fica:

```c
ret = FIELD_GET(GENMASK(9, 6), reg);
```

## Nem tudo deve virar FIELD_GET

Uma lição importante foi perceber que nem toda manipulação de bits precisa ser convertida.

Por exemplo:

```c
adis->tx[3] = value & 0xff;
```

Poderia ser escrito como:

```c
adis->tx[3] = FIELD_GET(GENMASK(7, 0), value);
```

Mas isso não necessariamente melhora a legibilidade.

Nesse caso, a intenção original — obter o byte menos significativo — já é bastante clara.

## Flags de um único bit

Outro padrão encontrado no driver `adxl372` foi:

```c
#define ADXL372_STATUS_1_FIFO_FULL(x) (((x) >> 2) & 0x1)
```

Inicialmente pensei em convertê-lo para `FIELD_GET()`, mas descobri que o estilo mais comum do kernel para flags booleanas é utilizar `BIT()`:

```c
#define ADXL372_STATUS_1_FIFO_FULL BIT(2)
```

E então verificar a flag diretamente:

```c
if (status1 & ADXL372_STATUS_1_FIFO_FULL)
```

Essa forma deixa claro que estamos apenas verificando a presença de um bit específico.

## Buscando candidatos para refatoração

Para encontrar possíveis conversões utilizei alguns comandos simples:

```bash
grep -rn ">>" drivers/iio/ | grep "&"
```

para localizar extrações manuais de campos.

E:

```bash
grep -rn "<<" drivers/iio/ | grep "|"
```

para encontrar possíveis candidatos a `FIELD_PREP()` ou `FIELD_MODIFY()`.

Também foi útil verificar quais drivers já utilizavam as macros modernas:

```bash
grep -rn "FIELD_GET\|FIELD_PREP\|FIELD_MODIFY" drivers/iio/
```
